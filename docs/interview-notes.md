# 面试讲稿 v0.1

## 1 分钟项目介绍

我做了一个 AI 应用安全审计实验平台，目标是模拟真实企业 RAG 和 Agent 应用中的安全问题。第一版先实现了用户注册登录、Bearer Token 鉴权、租户隔离、文档上传和文档归属控制。

这个阶段重点不是模型效果，而是先把 AI 应用的数据边界跑通：文档属于谁、属于哪个租户、谁能访问、越权访问是否会被拒绝。因为后续 RAG 检索和 Agent 工具调用都必须建立在这些权限边界上。

后续我会接入向量库和 RAG 查询，复现 Prompt Injection、RAG 污染、跨租户检索、系统提示泄露和 Agent 工具调用越权，并用 promptfoo、garak 或 PyRIT 做自动化安全测试。

## 项目亮点

- 没有只做普通聊天 Demo，而是从安全审计角度设计 AI 应用。
- 第一版先完成身份认证、权限边界和文档归属。
- 后续 RAG 检索必须继承 `owner_id` 和 `tenant_id` 控制。
- 项目可以自然扩展到 Prompt Injection、RAG 污染、工具调用越权。

## 面试官可能追问

### 为什么第一版没有直接接大模型？

因为 AI 应用安全不是只测提示词。真实风险往往发生在数据流和权限边界上。如果没有先设计用户、租户、文档归属，后续 RAG 接入后很容易出现跨用户检索和敏感信息泄露。因此第一版先做安全边界，再接模型能力。

### 如何验证越权？

注册 Alice 和 Bob 两个普通用户。Alice 上传文档后，Bob 使用自己的 Token 请求 `/documents/{alice_document_id}`。预期结果是返回 `403 Forbidden`，并且响应中不包含 Alice 的文档内容。

### 后续 RAG 会怎么做安全控制？

检索阶段必须带上当前用户的 `tenant_id` 和授权条件，只允许召回该用户可访问的文档。不能先召回所有文档再让模型判断是否可见，因为模型不能作为权限边界。
